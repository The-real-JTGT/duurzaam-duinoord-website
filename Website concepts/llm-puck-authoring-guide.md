# LLM Puck Authoring Guide

## Purpose

This document explains how the drag-and-drop website builder works in this project and how an LLM should generate new pages so they remain editable in the builder.

The short version:

- The editable website is powered by Puck plus Supabase.
- The source of truth for editable pages is Puck JSON data, not hardcoded page JSX.
- If an LLM should create editable pages, it should usually generate Puck-compatible `Data` objects or add new Puck blocks.

## Core Architecture

### Builder config

The central builder contract is [`frontend/src/puck.config.tsx`](C:/Users/RL51721/Documents/95%20Duurzaam%20Duinoord/New%20Website/DD-v0.1/frontend/src/puck.config.tsx).

It defines:

- which blocks exist
- which editor fields each block exposes
- default props for new block instances
- how each block renders
- which block categories appear in the editor sidebar
- the shared page wrapper via `root.render`

`root.render` wraps every rendered page with:

- [`Navbar`](C:/Users/RL51721/Documents/95%20Duurzaam%20Duinoord/New%20Website/DD-v0.1/frontend/src/components/Navbar.tsx)
- [`Footer`](C:/Users/RL51721/Documents/95%20Duurzaam%20Duinoord/New%20Website/DD-v0.1/frontend/src/components/Footer.tsx)

That means page content generated by Puck should only describe the body content. The site chrome is already added by the builder config.

### Editor flow

The editor route is [`frontend/src/app/edit/[...slug]/page.tsx`](C:/Users/RL51721/Documents/95%20Duurzaam%20Duinoord/New%20Website/DD-v0.1/frontend/src/app/edit/[...slug]/page.tsx).

Flow:

1. The route converts `/edit/foo/bar` into the slug `/foo/bar`.
2. It loads existing page content from the `pages` table in Supabase.
3. If no stored content exists, it falls back to [`getDefaultPageData`](C:/Users/RL51721/Documents/95%20Duurzaam%20Duinoord/New%20Website/DD-v0.1/frontend/src/lib/default-page-data.ts).
4. It renders the Puck editor with that data.
5. On publish, it upserts the new JSON back into Supabase.

### Public render flow

The main dynamic render route is [`frontend/src/app/[...slug]/page.tsx`](C:/Users/RL51721/Documents/95%20Duurzaam%20Duinoord/New%20Website/DD-v0.1/frontend/src/app/[...slug]/page.tsx).

Flow:

1. The route turns the URL into a slug such as `/mobiliteit`.
2. It reads `content` from Supabase for that slug.
3. It renders the page using `<Render config={config} data={page.content} />`.

This is the key rule:

> If the LLM wants the result to stay editable, it should generate data that Puck can render, rather than writing standalone JSX page markup.

### Storage

Supabase stores editable pages in the `pages` table defined in [`backend/supabase_schema.sql`](C:/Users/RL51721/Documents/95%20Duurzaam%20Duinoord/New%20Website/DD-v0.1/backend/supabase_schema.sql).

Relevant fields:

- `slug TEXT NOT NULL UNIQUE`
- `title TEXT NOT NULL`
- `content JSONB NOT NULL`

## Data Model Used In This Repo

This project uses Puck `Data` objects. The basic shape is:

```ts
{
  content: [
    {
      type: "Hero",
      props: {
        id: "hero-example",
        title: "Example",
        subtitle: "Example subtitle",
        buttonText: "Read more",
        buttonLink: "/example"
      }
    }
  ],
  root: {
    props: {
      title: "Example Page"
    }
  },
  zones: {}
}
```

### Important rules for generated data

- `content` is the ordered top-level page body.
- Every block instance should have a unique `props.id`.
- `type` must match a registered block name exactly.
- `props` must match the block's field schema.
- `root.props.title` should be present.
- `zones` is required for nested content such as grid children.
- Slugs must match the route shape exactly and should start with `/`.

## Common Page Patterns

### Pattern 1: Simple information page

Best for:

- topic pages
- explanation pages
- pages with mostly text and lists

Recommended structure:

1. `Hero`
2. `SectionHeader`
3. `Text`
4. `SectionHeader`
5. `BulletList`
6. `ContactCTA`

Template:

```ts

  content: [
    {
      type: "Hero",
      props: {
        id: "hero-info",
        title: "Page title",
        subtitle: "Short page introduction",
        buttonText: "Neem contact op",
        buttonLink: "mailto:info@duurzaamduinoord.nl"
      }
    },
    {
      type: "SectionHeader",
      props: {
        id: "section-1",
        title: "Wat is het?",
        subtitle: "",
        alignment: "left"
      }
    },
    {
      type: "Text",
      props: {
        id: "text-1",
        content: "Paragraph one.\\n\\nParagraph two."
      }
    },
    {
      type: "SectionHeader",
      props: {
        id: "section-2",
        title: "Wat doen wij al?",
        subtitle: "",
        alignment: "left"

    {
      type: "BulletList",
```

## Authoring Templates

### Template: Add a new seeded default page

Use this when a new slug should open with starter content in the editor.

1. Add a new `const <name>Data: Data = { ... }` object in [`frontend/src/lib/default-page-data.ts`](C:/Users/RL51721/Documents/95%20Duurzaam%20Duinoord/New%20Website/DD-v0.1/frontend/src/lib/default-page-data.ts).
2. Add the slug mapping to the `defaults` record in `getDefaultPageData`.

Template:

```ts
const voorbeeldData: Data = {
  content: [
    {
      type: "Hero",
      props: {
        id: "hero-voorbeeld",
        title: "Voorbeeldpagina",
        subtitle: "Introductie van de pagina.",
        buttonText: "Doe mee",
        buttonLink: "mailto:info@duurzaamduinoord.nl"
      }
    }
  ],
  root: { props: { title: "Voorbeeldpagina" } },
  zones: {}
};

export function getDefaultPageData(slug: string): Data {
  const defaults: Record<string, Data> = {
    "/voorbeeld": voorbeeldData
  };

  return defaults[slug] || { content: [], root: { props: { title: "" } }, zones: {} };
}
```

### Template: Add a new block

Use this when current blocks cannot express the needed design or content model cleanly.

Steps:

1. Create a new file in [`frontend/src/components/blocks`](C:/Users/RL51721/Documents/95%20Duurzaam%20Duinoord/New%20Website/DD-v0.1/frontend/src/components/blocks).
2. Export a prop type.
3. Export a `ComponentConfig<...>`.
4. Add the type to `Props` in [`frontend/src/puck.config.tsx`](C:/Users/RL51721/Documents/95%20Duurzaam%20Duinoord/New%20Website/DD-v0.1/frontend/src/puck.config.tsx).
5. Import and register the block in `components`.
6. Add it to a `category`.
7. Optionally seed it in [`default-page-data.ts`](C:/Users/RL51721/Documents/95%20Duurzaam%20Duinoord/New%20Website/DD-v0.1/frontend/src/lib/default-page-data.ts).

Template:

```tsx
import React from "react";
import { ComponentConfig } from "@measured/puck";

export type QuoteBlockProps = {
  quote: string;
  author: string;
};

export const QuoteBlock: ComponentConfig<QuoteBlockProps> = {
  fields: {
    quote: { type: "textarea" },
    author: { type: "text" }
  },
  defaultProps: {
    quote: "Plaats hier een quote.",
    author: "Naam"
  },
  render: ({ quote, author }) => {
    return (
      <section className="w-full py-12 px-4 md:px-8">
        <div className="max-w-3xl mx-auto">
          <blockquote className="text-2xl font-serif italic text-foreground">
            {quote}
          </blockquote>
          <p className="mt-4 text-foreground/70">{author}</p>
        </div>
      </section>
    );
  }
};
```

Then register it in `puck.config.tsx`:

```ts
type Props = {
  Quote: QuoteBlockProps;
};

categories: {
  content: {
    title: "Inhoud",
    components: ["Text", "BulletList", "ImageBanner", "Quote"]
  }
},

components: {
  Quote: QuoteBlock
}
```

## LLM Decision Rules

### Prefer existing blocks when

- the page can be expressed as stacked sections
- the page is mostly text, lists, images, cards, or a CTA
- the layout can be represented with `Grid`
- the content differences are editorial rather than structural

### Add a new block when

- editors need a new reusable content type
- the needed field structure cannot be represented cleanly with current props
- a page would otherwise require repeated awkward block combinations
- the layout or behavior should be reusable across multiple pages

### Avoid hardcoded page JSX when

- the page is meant to stay editable in Puck
- the content is not just a temporary fallback
- the goal is to let editors reorder or rewrite sections later

## Checklist For LLM-Generated Pages

### Content generation checklist

- [ ] Use Puck `Data`, not standalone JSX, for editable pages.
- [ ] Use exact registered block names.
- [ ] Give every block a unique `props.id`.
- [ ] Add `root.props.title`.
- [ ] Use `zones: {}` even when there are no nested blocks.
- [ ] For `BulletList`, provide newline-delimited `items`.
- [ ] For `FeatureCard`, use only `leaf`, `sun`, `recycle`, or `home`.
- [ ] For `SectionHeader`, use only `left` or `center`.
- [ ] For `ImageBanner`, use only `rectangle`, `arch`, or `rounded`.
- [ ] For `Grid`, store children in `zones["<grid-id>:grid-items"]`.

### New page checklist

- [ ] Decide whether the page should be seeded in `default-page-data.ts`, stored directly in Supabase, or both.
- [ ] Ensure the slug starts with `/`.
- [ ] Ensure the editor slug and render slug refer to the same page.
- [ ] Reuse block patterns already present in the project where possible.

### New block checklist

- [ ] Create a new block file under `src/components/blocks`.
- [ ] Export a typed props interface.
- [ ] Define editor `fields`.
- [ ] Define sensible `defaultProps`.
- [ ] Implement `render`.
- [ ] Register the block in `puck.config.tsx`.
- [ ] Add the block to a category.
- [ ] Optionally seed it in `default-page-data.ts`.
- [ ] Verify that the resulting content still renders through `<Render config={config} ... />`.

### Verification checklist

- [ ] Open the page in the editor.
- [ ] Confirm all fields appear in the sidebar.
- [ ] Reorder blocks and verify the page still renders correctly.
- [ ] Publish and confirm the content is saved to Supabase.
- [ ] Load the public slug and verify the rendered output.
- [ ] If using `Grid`, verify nested items appear in the correct zone.

## Known Repo Quirks

These are current project behaviors that an LLM should be aware of.

### Homepage slug mismatch

The homepage render path reads slug `/`, but the homepage edit button points to `/edit/home`.

Relevant files:

- [`frontend/src/app/page.tsx`](C:/Users/RL51721/Documents/95%20Duurzaam%20Duinoord/New%20Website/DD-v0.1/frontend/src/app/page.tsx)
- [`frontend/src/components/EditButton.tsx`](C:/Users/RL51721/Documents/95%20Duurzaam%20Duinoord/New%20Website/DD-v0.1/frontend/src/components/EditButton.tsx)

Implication:

- Editing `/edit/home` stores content under `/home`, while the real homepage reads `/`.
- An LLM should not assume homepage editing is fully aligned until this is fixed.

### Navbar edit link is placeholder

[`frontend/src/components/Navbar.tsx`](C:/Users/RL51721/Documents/95%20Duurzaam%20Duinoord/New%20Website/DD-v0.1/frontend/src/components/Navbar.tsx) currently links to `/edit/test`.

Implication:

- This is not a reliable example of the intended editor path.

### Published title is hardcoded

[`frontend/src/app/edit/[...slug]/page.tsx`](C:/Users/RL51721/Documents/95%20Duurzaam%20Duinoord/New%20Website/DD-v0.1/frontend/src/app/edit/[...slug]/page.tsx) currently saves `title: "New Page"` on publish.

Implication:

- `root.props.title` is useful for page structure, but database metadata is not yet synced from it.

### Empty page behavior

[`frontend/src/lib/get-puck-content.ts`](C:/Users/RL51721/Documents/95%20Duurzaam%20Duinoord/New%20Website/DD-v0.1/frontend/src/lib/get-puck-content.ts) returns `null` if `content.content.length === 0`.

Implication:

- A stored page with zero top-level blocks is treated like missing builder content.

## Recommended LLM Output Strategy

When asked to create a new editable page, prefer this order:

1. Reuse existing blocks.
2. Produce a valid Puck `Data` object.
3. Seed it in [`default-page-data.ts`](C:/Users/RL51721/Documents/95%20Duurzaam%20Duinoord/New%20Website/DD-v0.1/frontend/src/lib/default-page-data.ts) if the page should start with default content in the editor.
4. Add a new block only if the current block set is insufficient.

## Image Library Workflow

If a page needs images, do not default to random stock imagery when the Duurzaam Duinoord library is available.

Use this workflow:

1. Search the library before filling any image field.
2. Prefer the image library `resolvedUrl` over external image URLs.
3. Reuse the library `altText` when it is already suitable.
4. Prefer local assets from `frontend/public/media/library/`.
5. If the library contains multiple plausible matches, prefer the one whose source page or topic best matches the current page brief.
